# Over-Engineering Audit — `agent-refinement/skills`

Date: 2026-06-23
Scope: complexity / over-engineering only (correctness, security, and performance routed elsewhere).
Method: whole-tree scan (ponytail-audit). Lists findings; applies nothing.

## Verdict

This is a docs-first workshop repo. The committed Python (3 units:
`extract_opencode_transcript.py`, `ttlcache`, `wordstats`) is **lean** — the
boundary parsing in the extractor is justified deserialization validation, not
over-engineering. No dependency is removable: `uv`, `ruff`, `ty`, `pytest`,
`hypothesis`, `prek` are all used, and no stdlib is hand-rolled. **The bloat is
entirely documentary and in fixture duplication.**

### Caveat on #1 and #2

Both are **deliberate retention** per `PRD.md` §6 (source-of-truth map lists the
planning docs as "historical design rationale") and the fixture-snapshot design.
The audit's job is to name them as cuttable debt — the "kept for later, later
never comes" class. Acting on them is your decision, not a defect. This tension
is also visible against the repo's own non-goal #6 ("prefer small, boring,
reversible improvements"): the `model-tests/` doc set (9 files, ~1,100 lines) is
now larger than everything it documents.
